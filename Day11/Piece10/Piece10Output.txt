Day 11 — Profile a slow endpoint
=================================

## Baseline p50 / p99

Load: 2 virtual users, 20 iterations against GET /api/quotes/slow

  http_req_duration: avg=1.24s  min=1.01s  med=1.23s  max=1.41s  p(90)=1.40s  p(99)=1.41s

  p50 = 1.23 s
  p99 = 1.41 s

A healthy read endpoint should respond in under 50 ms.
This endpoint is ~25x slower than acceptable.

---

## The Offending SQL

One request to GET /api/quotes/slow emits 21 queries:

Query 1 — get all distinct authors:
  SELECT DISTINCT "q"."Author"
  FROM "Quotes" AS "q"
  WHERE NOT ("q"."IsDeleted")

Queries 2–21 — one per author (repeated 20 times with a different @author each time):
  SELECT "q"."Id", "q"."Author", "q"."CreatedByUserId", "q"."IsDeleted", "q"."Text"
  FROM "Quotes" AS "q"
  WHERE "q"."Author" = @author AND NOT ("q"."IsDeleted")

With 20 authors in the database, every single request fires 21 SQL queries
instead of 1. Under load this compounds — 20 concurrent users × 21 queries
= 420 simultaneous database round trips.

---

## Execution Plan

  SET SHOWPLAN_TEXT ON;
  GO
  SELECT Id, Author, Text FROM Quotes WHERE Author = 'Einstein' AND IsDeleted = 0;
  GO

  StmtText
  |--Clustered Index Scan(OBJECT:([QuotesDb].[dbo].[Quotes].[PK__Quotes__3214EC07]),
     WHERE:([QuotesDb].[dbo].[Quotes].[IsDeleted]=[@2]
     AND [QuotesDb].[dbo].[Quotes].[Author]=CONVERT_IMPLICIT(nvarchar(4000),[@1],0)))

Clustered Index Scan means SQL Server is reading every row in the Quotes table
to find rows where Author = 'Einstein'. There is no index on Author so the
database has no shortcut — it must scan all 500 rows on every per-author query.
A Clustered Index Scan is the most expensive operation — equivalent to a full
table scan. With an index on Author this would become a much cheaper
Index Seek, jumping directly to matching rows.

---

## Two Biggest Problems

Problem 1 — N+1 queries
  The endpoint fetches distinct authors in one query, then issues a separate
  query for each author inside a foreach loop. With 20 authors that is 21
  queries per request. The fix is a single grouped query:

    SELECT Author, Id, Text
    FROM Quotes
    WHERE IsDeleted = 0
    ORDER BY Author

  One query replaces 21. EF equivalent:
    db.Quotes
      .Where(q => !q.IsDeleted)
      .GroupBy(q => q.Author)
      .Select(g => new { Author = g.Key, Quotes = g.ToList() })

Problem 2 — Missing index on Author
  Every per-author WHERE clause performs a full table scan (Clustered Index Scan).
  Adding an index on Author drops this from O(n) to O(log n):

    CREATE INDEX IX_Quotes_Author ON Quotes (Author);

  EF Core migration equivalent:
    entity.HasIndex(q => q.Author);

  With the index, the execution plan changes from Clustered Index Scan to
  Index Seek, meaning SQL Server jumps directly to matching rows.
