1) Diagnosis Note :

N+1 Query — Diagnosis Note

When GET /api/quotes was called with 10 quotes in the DB, Jaeger showed 12 spans: 1 HTTP span + 1 query to fetch IDs + 10 separate FindAsync queries (one per quote). This is the N+1 problem — for every extra row, you pay an extra DB round trip.

Root cause: GetPagedAsync first fetched a list of IDs, then looped over them calling FindAsync individually instead of fetching all rows in one query.

Fix: Replaced the two-step loop with a single LINQ query using .Where().OrderBy().Skip().Take().ToListAsync(). EF Core translates this into one SQL with LIMIT/OFFSET.

Before: 12 spans, 11 DB queries for 10 quotes
After: 2 spans, 1 DB query for 10 quotes

Key lesson: N+1 looks fine in code review — you only see it in traces. Jaeger makes it obvious as a waterfall of identical DB spans. FindAsync inside a loop is always a red flag.
