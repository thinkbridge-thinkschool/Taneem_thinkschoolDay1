Day 10 — Query translation + projections
=========================================

## Original SQL (whole entity)

Query:
var quotes = await db.Quotes
    .Where(q => q.Author == "Einstein")
    .ToListAsync();

SQL generated:
SELECT "q"."Id", "q"."Author", "q"."CreatedAt", "q"."Text"
FROM "Quotes" AS "q"
WHERE "q"."Author" = 'Einstein'

EF fetches every column even if the caller only needs two of them.

---

## Projected query + leaner SQL

Query:
var quotes = await db.Quotes
    .Where(q => q.Author == "Einstein")
    .Select(q => new QuoteDto { Author = q.Author, Text = q.Text })
    .ToListAsync();

SQL generated:
SELECT "q"."Author", "q"."Text"
FROM "Quotes" AS "q"
WHERE "q"."Author" = 'Einstein'

.Select() tells EF exactly which columns are needed.
Id and CreatedAt are gone — the database sends less data across the wire.

---

## Accidental client-side evaluation — caught and fixed

### BAD (accidental IEnumerable type leak)

IEnumerable<Quote> source = db.Quotes.Where(q => q.Author == "Einstein");
var quotes = source.Where(q => q.Text.Length > 40).ToList();

SQL generated:
SELECT "q"."Id", "q"."Author", "q"."CreatedAt", "q"."Text"
FROM "Quotes" AS "q"
WHERE "q"."Author" = 'Einstein'

The developer assigned to IEnumerable<Quote> instead of IQueryable<Quote>.
That silently lost the queryable — the second .Where() became LINQ-to-Objects.
The text length filter never reached the database; it ran in C# after all
Einstein rows were already loaded into memory.
No error, no warning — wrong SQL, correct result.

### FIXED

var quotes = await db.Quotes
    .Where(q => q.Author == "Einstein")
    .Where(q => q.Text.Length > 40)
    .ToListAsync();

SQL generated:
SELECT "q"."Id", "q"."Author", "q"."CreatedAt", "q"."Text"
FROM "Quotes" AS "q"
WHERE "q"."Author" = 'Einstein' AND length("q"."Text") > 40

Staying on IQueryable keeps both filters in SQL.
Only rows that match both conditions cross the wire.
