Day 12 — When to reach for Dapper
===================================

## EF Core Implementation

File: Queries/GetQuotesSummaryHandler.cs

  public async Task<List<QuoteSummaryReadModel>> HandleAsync(
      GetQuotesSummaryQuery query, CancellationToken ct)
  {
      var rows = await _db.Quotes
          .Where(q => !q.IsDeleted)
          .OrderBy(q => q.Id)
          .Skip((query.Page - 1) * query.Size)
          .Take(query.Size)
          .Select(q => new { q.Id, q.Author, q.Text })
          .ToListAsync(ct);

      return rows.Select(q => new QuoteSummaryReadModel(
          q.Id,
          q.Author,
          string.Concat(q.Author.Split(' ').Where(w => w.Length > 0).Select(w => w[0])),
          q.Text.Length > 100 ? q.Text[..100] + "…" : q.Text
      )).ToList();
  }

SQL generated by EF:
  SELECT [q].[Id], [q].[Author], [q].[Text]
  FROM [Quotes] AS [q]
  WHERE [q].[IsDeleted] = 0
  ORDER BY [q].[Id]
  OFFSET @Offset ROWS FETCH NEXT @Size ROWS ONLY

Endpoint: GET /api/quotes/summary?page=1&size=10

## Dapper Implementation

File: Queries/GetQuotesSummaryDapperHandler.cs

  public async Task<IEnumerable<QuoteSummaryReadModel>> HandleAsync(
      GetQuotesSummaryQuery query, CancellationToken ct)
  {
      const string sql = """
          SELECT Id, Author, Text
          FROM Quotes
          WHERE IsDeleted = 0
          ORDER BY Id
          OFFSET @Offset ROWS FETCH NEXT @Size ROWS ONLY
          """;

      using var connection = new SqlConnection(_connectionString);

      var rows = await connection.QueryAsync<(int Id, string Author, string Text)>(
          sql,
          new { Offset = (query.Page - 1) * query.Size, query.Size });

      return rows.Select(q => new QuoteSummaryReadModel(
          q.Id,
          q.Author,
          string.Concat(q.Author.Split(' ').Where(w => w.Length > 0).Select(w => w[0])),
          q.Text.Length > 100 ? q.Text[..100] + "…" : q.Text
      ));
  }

Endpoint: GET /api/quotes/summary/dapper?page=1&size=10

Load: 2 virtual users, 100 iterations each

  EF Core (GET /api/quotes/summary):
    avg=125ms  min=43ms  med=98ms  max=458ms  p(90)=207ms  p(99)=437ms
    throughput: 15 req/s

  Dapper (GET /api/quotes/summary/dapper):
    avg=21ms   min=2ms   med=7ms   max=642ms  p(90)=23ms   p(99)=260ms
    throughput: 59 req/s

  Improvement:
    p50 (med): 98ms ÷ 7ms   = 14x faster
    p99:       437ms ÷ 260ms = 1.7x faster
    throughput: 59 ÷ 15      = 4x more requests per second

## The SQL

Both implementations emit the same SQL:
  SELECT Id, Author, Text
  FROM Quotes
  WHERE IsDeleted = 0
  ORDER BY Id
  OFFSET N ROWS FETCH NEXT N ROWS ONLY

The difference is not the SQL — it is the overhead around the SQL.
EF builds an expression tree, compiles it to SQL, tracks entities,
and maps results through a model. Dapper sends raw SQL and maps
results directly to a struct — no tracking, no compilation, no overhead.

---

## Rule for When to Use Dapper vs EF

Use EF Core by default. It handles database changes, relationships, validation, and tracking automatically, which is enough for most APIs.

Use Dapper only for very fast read operations where testing shows that EF Core is too slow. Examples include frequently used list endpoints, large reports, or dashboard queries that read a lot of data.

In this case, Dapper was 14 times faster because the query only reads data and does not need change tracking or relationships.

If the feature needs to update data or work with related entities, EF Core is the better choice.

Keep writes (INSERT, UPDATE, DELETE) in EF Core and use Dapper only for performance-critical reads.
